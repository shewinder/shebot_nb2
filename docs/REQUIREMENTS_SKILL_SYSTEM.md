# SheBot SKILL 系统需求文档

## 1. 概述

### 1.1 背景
SheBot 需要支持通过 SKILL 机制扩展 AI 能力。SKILL 是一种声明式的能力扩展方式，允许用户通过编写 Markdown 文件和脚本来为 AI 添加新功能，而无需修改核心代码。

### 1.2 目标
- 用户可以通过放置 SKILL 文件到指定目录来扩展 AI 能力
- AI 能够根据需要自动发现和激活合适的 SKILL
- 系统支持渐进式加载，避免上下文膨胀
- 提供细粒度的权限控制，确保安全性

### 1.3 术语定义
| 术语 | 定义 |
|------|------|
| SKILL | 包含描述文件和可选脚本的目录，用于扩展 AI 能力 |
| SKILL.md | 每个 SKILL 的核心描述文件，包含元数据和指导说明 |
| 元工具 | 系统提供的用于管理 SKILL 的特殊工具 |
| 内置工具 | 系统提供的基础能力工具（如执行命令、读写文件等） |
| 渐进式披露 | 仅加载必要信息到上下文的机制 |

---

## 2. 功能需求

### 2.1 SKILL 发现与加载

#### FR-001 自动扫描
系统应在启动时自动扫描配置的 SKILL 目录，发现所有有效的 SKILL。

#### FR-002 SKILL 识别
系统应识别包含 SKILL.md 文件的目录为一个 SKILL。

#### FR-003 元数据提取
系统应从 SKILL.md 的 YAML frontmatter 中提取以下元数据：
- 名称（唯一标识）
- 一句话描述
- 允许使用的工具列表
- 是否允许 AI 自动触发
- 是否允许用户手动触发

#### FR-004 错误处理
当 SKILL.md 格式错误或缺少必填字段时，系统应记录警告并跳过该 SKILL，不影响其他 SKILL 的加载。

---

### 2.2 渐进式披露

#### FR-005 元数据层暴露
系统应将发现的 SKILL 列表（仅名称和描述）暴露给 AI，使 AI 能够了解有哪些 SKILL 可用。

#### FR-006 按需激活
AI 应根据用户意图选择合适的 SKILL，系统支持 AI 通过调用特定功能来激活 SKILL。

#### FR-007 内容注入
当 SKILL 被激活后，系统应将其 SKILL.md 的正文内容注入到对话上下文中，供 AI 阅读和理解。

#### FR-008 已激活标记
系统应跟踪当前会话中已激活的 SKILL，避免重复激活。

#### FR-009 内容隔离
注入的 SKILL 内容应对用户可见或隐藏可配置，但 AI 必须能够阅读。

---

### 2.3 工具系统

#### FR-010 元工具提供
系统应提供以下元工具：
- **activate_skill**: 激活指定 SKILL
- **list_skills**: 列出所有可用 SKILL 及其基本信息

#### FR-011 内置工具集
系统应提供以下内置工具供 SKILL 使用：
- **Bash**: 执行 shell 命令
- **Read**: 读取文件内容
- **Write**: 写入文件内容
- **Grep**: 文本搜索

#### FR-012 工具权限控制
每个 SKILL 应能声明允许使用的工具列表。系统应在执行工具前验证该 SKILL 是否有权限使用。

#### FR-013 权限拒绝处理
当 SKILL 尝试使用未授权的工具时，系统应返回明确的权限错误信息。

---

### 2.4 脚本执行

#### FR-014 脚本目录
SKILL 可以在 `scripts/` 目录下放置可执行脚本。

#### FR-015 脚本调用
AI 应通过内置工具（如 Bash）调用脚本，而不是直接作为 function calling 工具。

#### FR-016 参数传递
系统应支持将参数传递给脚本，参数可以通过命令行或临时文件传递。

#### FR-017 执行隔离
脚本应在受限环境中执行，系统应实施以下限制：
- 执行超时控制
- 资源使用限制（内存、CPU）
- 路径限制（防止访问 SKILL 目录外）

#### FR-018 结果返回
脚本的输出结果应能被 AI 读取和解析，支持文本和结构化数据（如 JSON）。

---

### 2.5 用户交互

#### FR-019 显式激活命令
用户应能通过特定命令格式（如 `#使用 <skill-name>`）显式激活 SKILL。

#### FR-020 激活提示
当 SKILL 被激活时，系统应向用户发送提示消息，告知 SKILL 已就绪。

#### FR-021 当前 SKILL 查询
用户应能查询当前会话中已激活的 SKILL 列表。

#### FR-022 SKILL 停用
用户应能停用已激活的 SKILL，系统应清理相关的上下文内容。

---

### 2.6 配置管理

#### FR-023 搜索路径配置
管理员应能配置 SKILL 的搜索路径，支持多个路径和通配符。

#### FR-024 启用/禁用开关
管理员应能全局启用或禁用 SKILL 系统。

#### FR-025 默认权限配置
管理员应能配置 SKILL 的默认允许工具列表。

---

## 3. 非功能需求

### 3.1 性能

#### NFR-001 启动时间
SKILL 扫描应在系统启动时完成，单个 SKILL 的解析时间不应超过 100ms。

#### NFR-002 上下文大小
元数据层（所有 SKILL 的名称和描述）不应超过 15000 字符。

#### NFR-003 激活延迟
SKILL 从激活到可用的时间不应超过 500ms。

### 3.2 安全性

#### NFR-004 权限最小化
SKILL 默认不应有任何权限，必须通过声明明确授权。

#### NFR-005 路径限制
SKILL 脚本只能访问其所在目录及其子目录。

#### NFR-006 命令限制
当 SKILL 声明了 `Bash(python scripts/*)` 时，系统应确保只能执行 scripts 目录下的 python 脚本。

#### NFR-007 敏感信息保护
环境变量中的敏感信息（如 API keys）不应传递给 SKILL 脚本，除非显式配置。

### 3.3 可靠性

#### NFR-008 错误隔离
单个 SKILL 的加载失败或执行失败不应影响系统或其他 SKILL 的正常运行。

#### NFR-009 优雅降级
当 SKILL 系统出现故障时，核心对话功能应能正常工作。

### 3.4 可维护性

#### NFR-010 日志记录
系统应记录以下事件到日志：
- SKILL 加载成功/失败
- SKILL 激活/停用
- 工具权限拒绝
- 脚本执行超时或异常

#### NFR-011 调试支持
系统应提供调试模式，输出详细的 SKILL 加载和调用信息。

---

## 4. SKILL 格式规范

### 4.1 目录结构

```
skills/
└── <skill-name>/
    ├── SKILL.md              # 必需：描述文件
    ├── scripts/              # 可选：可执行脚本
    │   └── *.py, *.sh
    ├── references/           # 可选：参考资料
    │   └── *.md, *.txt
    └── assets/               # 可选：静态资源
        └── templates/
```

### 4.2 SKILL.md 格式

#### 4.2.1 Frontmatter 部分
位于文件开头，用 `---` 包围，YAML 格式：

```yaml
---
name: skill-identifier          # 必需：唯一标识
description: 一句话描述         # 必需：用于 AI 选择
allowed-tools:                  # 可选：允许的工具列表
  - "Bash(python scripts/*)"
  - "Read"
user-invocable: true            # 可选：用户是否可手动触发（默认 true）
disable-model-invocation: false # 可选：是否禁止 AI 自动触发（默认 false）
---
```

#### 4.2.2 正文部分
Frontmatter 之后的内容，Markdown 格式，用于指导 AI 如何使用该 SKILL。

### 4.3 必需字段

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| name | string | 是 | 唯一标识符，小写字母、数字、连字符 |
| description | string | 是 | 一句话描述，用于 AI 决策 |
| allowed-tools | array | 否 | 允许使用的工具列表，默认继承系统配置 |
| user-invocable | boolean | 否 | 用户是否可通过命令触发，默认 true |
| disable-model-invocation | boolean | 否 | 是否禁止 AI 自动触发，默认 false |

---

## 5. 用户场景

### 5.1 场景一：自动激活

**用户输入**: "帮我计算 15 乘以 23 等于多少"

**系统行为**:
1. AI 看到可用的 SKILL 列表中有 "calculate"（计算器）
2. AI 判断用户需要数学计算，自动激活 calculate SKILL
3. 系统注入 calculate/SKILL.md 到上下文
4. AI 阅读 SKILL.md，了解如何调用 calculate.py 脚本
5. AI 使用 Bash 工具执行脚本
6. AI 解析结果并回复用户

### 5.2 场景二：手动激活

**用户输入**: "#使用 weather"

**系统行为**:
1. 系统激活 weather SKILL
2. 向用户发送提示："天气查询 SKILL 已激活"
3. 用户输入："北京今天天气"
4. AI 根据 SKILL.md 指导，调用天气查询 API
5. AI 回复天气信息

### 5.3 场景三：权限拒绝

**用户输入**: "帮我格式化硬盘"

**系统行为**:
1. AI 激活 system-admin SKILL（假设存在）
2. AI 尝试调用 Bash("rm -rf /")
3. 系统检查权限：system-admin 声明了 `allowed-tools: ["Read", "Write"]`，没有 Bash
4. 系统拒绝执行，返回权限错误
5. AI 告知用户无法执行此操作

### 5.4 场景四：脚本执行

**用户输入**: "分析这个目录下的代码复杂度"

**系统行为**:
1. AI 激活 code-analyzer SKILL
2. AI 阅读 SKILL.md，了解到需要使用 scripts/analyze.py
3. AI 构造命令：`python scripts/analyze.py --path ./src`
4. 系统验证：路径在 SKILL 目录内，符合权限
5. 系统执行脚本，60 秒后超时保护
6. AI 读取脚本输出的 JSON 报告，向用户展示结果

## 6. 边界情况

### 6.1 SKILL 冲突
当多个 SKILL 名称相同时，系统应使用后加载的 SKILL 或记录警告。

### 6.2 循环激活
当 SKILL A 指导 AI 激活 SKILL B，而 SKILL B 又指导激活 SKILL A 时，系统应检测并阻止无限循环。

### 6.3 工具数量超限
当激活的 SKILL 过多，导致元数据超过上下文限制时，系统应提示用户或自动停用最早激活的 SKILL。

### 6.4 脚本执行超时
当脚本执行超过配置的超时时间时，系统应终止进程并返回超时错误。

### 6.5 脚本返回错误
当脚本返回非零退出码或错误输出时，AI 应能够理解错误并向用户解释。

## 7. 验收标准

### 7.1 基础功能
- [ ] 系统能正确扫描并加载指定目录下的所有 SKILL
- [ ] AI 能够通过元工具了解有哪些 SKILL 可用
- [ ] AI 能够激活 SKILL 并获取其指导内容
- [ ] 激活的 SKILL 能在当前会话中持续有效

### 7.2 工具执行
- [ ] SKILL 能通过声明控制允许使用的工具
- [ ] 系统能在执行前验证工具权限
- [ ] 无权限的工具调用会被拒绝并返回错误
- [ ] 脚本能被正确执行，输出能被 AI 获取

### 7.3 用户体验
- [ ] 用户能通过命令显式激活 SKILL
- [ ] SKILL 激活时用户能收到提示
- [ ] 用户能查看当前激活的 SKILL 列表
- [ ] 用户能停用已激活的 SKILL

### 7.4 安全性
- [ ] SKILL 只能访问其目录内的文件
- [ ] 脚本执行受超时限制
- [ ] 环境变量中的敏感信息不会泄露给脚本
- [ ] 错误的 SKILL 不会影响系统稳定性
